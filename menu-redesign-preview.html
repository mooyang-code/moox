MooX 前端交互再设计 · 预览

目标：收敛「数据资产」子菜单、重构「数据列表」，从用户视角降低理解与使用成本。

现状：数据资产下 9 个平铺子菜单 数据来源数据对象数据集 字段管理因子管理查询视图 数据概览数据列表（4 层 tab）数据同步
三个方案的共同改进： ① 「数据概览」变成板块首页（点"数据资产"即落地），不再单列； ② 旧「数据列表」重构为「数据浏览」——先选数据集/视图，系统自动判断 时序/记录 与 主存/视图， 用户不用再手点 主存读取 / View查询 × TimeSeries / Record 四层 tab。
推荐 · 平衡

方案 A · 对象分组

字段+因子合并为"建模元素"一页；保留各对象独立页，学习曲线平缓。

数据资产 首页=概览仪表盘
├─ 数据来源
├─ 数据对象
├─ 数据集 列/对象绑定在页内 tab
├─ 建模元素 字段 + 因子 两 tab
├─ 查询视图
├─ 数据浏览 选对象→自动判类型
└─ 数据导入
子项：9 → 7
最精简 · 改动最大

方案 B · 阶段分组

建模收进一个"左树+右详情"工作台；数据页合并浏览+导入。

数据资产 首页=概览仪表盘
├─ 数据建模 左树:来源→对象→数据集
    右:列/字段/因子/绑定
├─ 查询视图
└─ 数据 浏览 / 导入 两 tab
子项：9 → 4
最小改动

方案 C · 轻收敛

仅概览变首页、浏览+导入合一；字段/因子各自保留。

数据资产 首页=概览
├─ 数据来源
├─ 数据对象
├─ 数据集
├─ 字段管理
├─ 因子管理
├─ 查询视图
└─ 数据 浏览 + 导入
子项：9 → 6

核心改造：「数据列表」→「数据浏览」

把"底层协议选择"交给系统推断，用户只面对自己懂的概念：数据集 / 视图。

现状 理解成本高

主存读取View 查询
TimeSeriesRecord
dataset_id　手填字符串
start_time　手填 RFC3339
dimensions　手填 JSON
order　SORT_ORDER_ASC（裸枚举）

用户需先理解 主存 vs 视图、时序 vs 记录，再手填底层字段。

重构后 零协议概念

第 1 步　选择数据集或视图（下拉，带名称搜索）
系统自动识别：该对象是 时序 还是 记录，是查 主存 还是 视图 → 自动切换对应表单
第 2 步　按对象类型只显示相关筛选项
（时序：对象/周期/时间范围；记录：record_id/版本）
时间/枚举　日期选择器 + 中文下拉（升序/降序），不暴露 RFC3339/裸枚举
▼ 一键查询，结果表格自适应列

顶部空间选择器 + 新建按钮

下拉内置"新建空间"项，下拉旁保留齿轮进入空间管理。
