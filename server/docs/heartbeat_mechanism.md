# 云函数心跳机制设计文档

## 概述

为解决云函数被自动迁移后任务丢失的问题，我们设计了一套心跳机制，实现云函数与Moox管理服务之间的状态同步。

## 设计原则

1. **快速检测**：5秒心跳间隔，11秒超时检测
2. **避免风暴**：随机延迟0-2秒，心跳聚合批处理
3. **增量同步**：只在任务变化时上报完整列表
4. **自动恢复**：检测到节点离线后自动尝试恢复

## 架构组件

### 1. 云函数端

- **HeartbeatService**：负责定时发送心跳
- **心跳数据**：包含节点ID、任务版本、任务哈希、活跃任务列表
- **恢复处理**：接收并处理Moox发送的恢复请求

### 2. Moox服务端

- **HeartbeatManager**：管理所有节点的心跳状态
- **HeartbeatAggregator**：聚合心跳请求，批量处理
- **节点监控**：定时检查节点健康状态
- **自动恢复**：探测离线节点并尝试恢复

## 数据流程

### 1. 正常心跳流程

```
云函数 ---(5s+随机延迟)---> Moox服务
         心跳数据
                            
Moox服务 ----------------> 云函数
         同步响应(如需要)
```

### 2. 节点恢复流程

```
1. 检测到节点离线（11秒无心跳）
2. 等待5秒二次确认
3. 主动调用云函数恢复接口
4. 同步任务列表到云函数
5. 云函数重启任务并立即心跳确认
```

## 配置说明

### 云函数配置

```yaml
runtime:
  serverless:
    node_id: "${SCF_FUNCTION_NAME}-${SCF_INSTANCE_ID}"
    moox_url: "https://moox.example.com"
```

### Moox服务配置

心跳管理器在服务启动时自动初始化，无需额外配置。

## 监控指标

1. **节点在线率**：在线节点数/总节点数
2. **心跳延迟**：心跳往返时间
3. **恢复成功率**：成功恢复次数/总恢复次数
4. **任务同步准确率**：同步成功的任务数/总任务数

## 故障处理

### 场景1：网络抖动

- 短暂的网络中断不会触发恢复流程
- 11秒超时 + 5秒确认时间提供足够的容错

### 场景2：云函数迁移

- 新实例启动后立即发送心跳
- Moox检测到恢复请求，同步任务列表
- 云函数根据任务列表恢复执行

### 场景3：Moox服务重启

- 云函数继续发送心跳
- Moox重启后自动恢复心跳处理
- 首次心跳携带完整任务列表

## 性能优化

1. **心跳聚合**：批量处理心跳，减少数据库操作
2. **增量更新**：只在变化时发送完整数据
3. **内存缓存**：节点状态缓存在内存中
4. **异步处理**：离线检测和恢复异步执行

## 扩展性

- 支持自定义心跳间隔和超时时间
- 可扩展心跳数据内容（如系统指标）
- 支持多种云函数平台
- 可集成监控告警系统
