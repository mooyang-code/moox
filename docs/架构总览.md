## 1. 项目概述

MooX 是一个面向量化金融场景的**一站式数据平台**，涵盖从多源数据采集、统一存储管理、到前端可视化交互的完整技术栈。项目采用 Go Monorepo 架构，通过 `go.work` 管理 9 个独立的 Go Module，配合 Vue 3 前端，构建出可单机部署也可分布式扩展的系统。

### 核心能力

| 能力域 | 说明 |
|--------|------|
| 多源数据采集 | 支持交易所（Binance 等）K 线、标的元数据采集，可扩展至行情、舆情等数据源 |
| 统一存储引擎 | 时序数据（K 线）、记录数据（标的信息）的统一存储与查询，支持列级更新 |
| 异步派生视图 | 写入主存后异步构建 OLAP 物化视图（DuckDB）和全文索引（Bleve） |
| 管理控制台 | 用户管理、权限控制、采集规则配置、包管理、云节点管理、监控 |
| 前端工作台 | 数据浏览、采集器管理、策略配置、运维监控的可视化界面 |

## 2. Monorepo 组织结构

项目使用 Go 1.24 + `go.work` 管理 Monorepo，所有 Go Module 统一存放在 `modules/` 目录下。

### Workspace Module 清单

| Module | 路径 | 说明 |
|--------|------|------|
| `moox/modules/account` | `modules/account` | 账户/用户模块 |
| `moox/modules/cli` | `modules/cli` | 命令行工具 |
| `moox/modules/collector` | `modules/collector` | 数据采集器 |
| `moox/modules/admin` | `modules/admin` | 管理服务（网关 + 业务逻辑） |
| `moox/modules/admin/proto/admingen` | `modules/admin/proto/admingen` | Admin proto 生成代码 |
| `moox/modules/factor` | `modules/factor` | 因子计算模块 |
| `moox/modules/order` | `modules/order` | 订单/交易模块 |
| `moox/modules/storage` | `modules/storage` | 存储引擎 |
| `moox/modules/storage/proto/gen` | `modules/storage/proto/gen` | Storage proto 生成代码 |

### Monorepo 规范

- **跨模块引用**：模块间只允许通过 `proto/` 生成的代码包（`storagepb`、`mooxpb`）进行依赖，不直接 import 对方内部实现
- **统一构建入口**：根 `Makefile` 转发至 `scripts/` 下的脚本执行复杂逻辑
- **独立二进制产物**：`bin/` 目录存放各模块编译产物（`moox-admin`、`moox-storage`、`moox-collector` 等）

## 4. 核心模块详解

### 4.1 Admin 模块（管理服务）

**入口**：`modules/admin/cmd/moox-admin/main.go`

Admin 模块是整个系统的**控制面核心**，承担统一网关、用户管理、采集配置、包管理、云节点管理、监控等职责。

#### 架构分层

```
modules/admin/
├── cmd/moox-admin/main.go       # 入口：bootstrap.Initialize() + trpc.NewServer()
├── internal/
│   ├── bootstrap/                 # 启动编排（初始化、依赖注入、TRPC 注册）
│   ├── gateway/                   # 统一 HTTP 网关（路由分发 + 鉴权 + 限流）
│   ├── service/                   # 业务服务层
│   │   ├── auth/                  # 认证鉴权（JWT + 盐值挑战 + HMAC）
│   │   ├── database/              # 数据库管理（SQLite + GORM）
│   │   ├── packagemgr/            # 包管理（COS 上传下载）
│   │   ├── collectmgr/            # 采集管理（规则下发 + 心跳接收）
│   │   ├── cloudnode/             # 云节点管理
│   │   ├── asynctask/             # 异步任务
│   │   └── monitor/               # 监控配置
│   └── ...
├── proto/                         # Proto 定义（admin/collect/infra/ops_service）
│   └── admingen/                  # 生成代码（独立 module）
├── schema/admin.sql               # 数据库 Schema（19 张表）
├── config/                        # 配置文件（gateway.yaml / trpc_go.yaml）
└── docs/                          # 模块文档（已合并到根 docs/admin/）
```

#### TRPC 服务定义

Admin 模块通过 4 个 proto 文件定义了 8 个 TRPC 服务：

| Proto 文件 | 服务 | 核心方法 |
|-----------|------|---------|
| `admin.proto` | AuthService | Login, Logout, RefreshToken, GetUserInfo, ChangePassword |
| | UserManageService | CreateUser, ListUsers, UpdateUser, DeleteUser |
| | RoleManageService | CreateRole, ListRoles, AssignRoles |
| `collect.proto` | CollectService | GetCollectRules, UpsertCollectRule, DeleteCollectRule |
| | CollectNodeService | RegisterNode, Heartbeat, GetNodeStatus |
| `infra.proto` | PackageService | UploadPackage, ListPackages, DownloadPackage |
| | CloudNodeService | RegisterCloudNode, ListCloudNodes, DeployToNode |
| `ops_service.proto` | AsyncTaskService | CreateTask, GetTaskStatus, CancelTask |
| | MonitorService | GetMetrics, GetAlerts, UpdateMonitorConfig |

#### 数据库模型

Admin 使用 SQLite + GORM，`schema/admin.sql` 定义了 19 张表：

- **认证体系**：`users`、`roles`、`user_roles`、`permissions`、`role_permissions`
- **采集管理**：`collect_rules`、`collect_nodes`、`collect_tasks`、`collect_node_tasks`
- **包管理**：`packages`、`package_versions`
- **云节点**：`cloud_nodes`、`cloud_node_deployments`
- **异步任务**：`async_tasks`、`async_task_logs`
- **系统配置**：`system_configs`、`monitor_configs`、`audit_logs`、`login_sessions`

#### 鉴权机制（三层）

1. **JWT 鉴权**（用户侧）：登录后发放 JWT Token，网关层校验有效性
2. **盐值挑战 + HMAC 签名**（内部服务）：采集器等内部组件通过 HMAC 签名通信
3. **裸 HTTP 内部校验**：部分内部接口使用 IP 白名单 + 共享密钥

### 4.2 Storage 模块（存储引擎）

**入口**：`modules/storage/cmd/moox-storage/main.go`

Storage 模块是系统的**数据面核心**，提供量化金融数据的统一存储与查询能力。

#### 核心设计理念

| 概念 | 说明 |
|------|------|
| **Space** | 顶层命名空间，实现数据隔离 |
| **Dataset** | 数据集，一个 Dataset 只绑定一个 DataSource |
| **DataSource** | 数据源（替代传统 Exchange 概念），描述数据来源 |
| **Subject** | 数据主体（如交易标的），不归属于 DataSource |
| **SubjectSymbol** | 外部代码映射（如 `BINANCE:BTCUSDT` → 统一 Subject） |
| **Factor** | 因子定义，不拆分 Def/Instance |
| **View** | 异步物化只读查询入口 |

#### 架构分层

```
modules/storage/
├── cmd/moox-storage/main.go      # 入口
├── internal/
│   ├── access/                    # Access Service（唯一公开入口，Facade 模式）
│   ├── store/                     # PrimaryStore（Pebble KV 主存）
│   ├── view/                      # DataView（派生读模型）
│   │   ├── timeseries/            # 时序视图（DuckDB 物化）
│   │   └── record/                # 记录视图（Bleve 全文索引）
│   ├── deriver/                   # 异步派生器（消费事件 → 构建视图）
│   ├── eventbus/                  # 事件总线（NATS）
│   ├── metadata/                  # 元数据管理（SQLite 控制面）
│   ├── router/                    # 路由解析（水平分片支持）
│   ├── schema/                    # Schema 管理
│   └── device/                    # 设备/存储引擎管理
├── proto/                         # 6 个 proto 文件
│   └── gen/                       # 生成代码（独立 module）
└── docs/                          # 模块文档（已合并到根 docs/storage/）
```

#### 数据模型

Storage 将所有数据统一为**事实行**模型：

```
事实行 = space_id + dataset_id + data_key + version
```

| 数据类型 | Key 前缀 | Key 组成 | Version |
|----------|----------|----------|---------|
| TimeSeries（时序） | `t\|` | `subject \| freq \| dimhash` | `data_time` |
| Record（记录） | `r\|` | `record_id` | `record_version` |

**写入语义**：列级 patch/upsert，NULL 值不覆盖已有数据。

#### 存储引擎分工

```
                    ┌──────────────────────────────────────────────┐
                    │              Access Service                   │
                    │       (唯一公开入口 / Facade)                  │
                    └───────┬──────────────┬───────────────────────┘
                            │              │
                    写入 ▼              查询 ▼
                    ┌──────────┐    ┌──────────────────────────┐
                    │  Pebble   │    │    DataView (读模型)      │
                    │ (主存KV)  │    │  ┌────────┐  ┌────────┐ │
                    └─────┬────┘    │  │DuckDB  │  │ Bleve  │ │
                          │         │  │(时序)  │  │(全文)  │ │
                    事件 ▼         │  └────────┘  └────────┘ │
                    ┌──────────┐    └──────────────────────────┘
                    │   NATS    │           ▲
                    │ (事件总线)│           │ 物化构建
                    └─────┬────┘    ┌───────┴───────┐
                          │         │   Deriver     │
                          └────────▶│  (异步派生器)  │
                                    └───────────────┘

    ┌──────────┐                              ┌──────────┐
    │  SQLite  │  (元数据控制面)               │ Parquet  │ (冷归档)
    └──────────┘                              └──────────┘
```

#### CQRS + 异步派生

Storage 采用 **CQRS（命令查询职责分离）** 模式：

- **写模型**：Pebble KV 存储，所有写入通过 Access Service 统一入口
- **读模型**：DuckDB 物化视图（时序 OLAP）+ Bleve 全文索引（记录检索）
- **派生流程**：写入 Pebble → 发布事件到 NATS → Deriver 消费事件 → 通过 Access 回读完整行 → 写入派生存储
- **幂等保证**：Deriver 消费幂等，且 Access 始终是唯一数据入口

#### 版本化物化视图

采用 **Blue-Green** 模式管理视图版本：

1. `view_version` 递增触发后台构建 `building_result`
2. 构建期间增量写入同时写 `active_result` + `building_result`
3. 构建完成后原子切换 `active_result` ← `building_result`

#### 运行角色

Storage 支持通过 `storage.roles` 配置三种运行角色：

| 角色 | 职责 | 默认部署 |
|------|------|----------|
| `access` | 对外服务，接收读写请求 | 单机模式默认启用 |
| `primary` | Pebble 主存节点 | 分布式模式独立部署 |
| `deriver` | 异步派生器，构建视图 | 单机模式默认内嵌 |

#### 水平分片

`router.Resolver` 支持多节点 Pebble 分片：
- 精确 `subject_id` 匹配 → 优先路由
- Glob pattern 匹配 → 次优先
- 通配 `*` → 兜底路由

### 4.3 Collector 模块（数据采集器）

**入口**：`modules/collector/cmd/moox-collector/main.go`

Collector 是**多源量化数据采集器**，设计为部署在腾讯云 SCF（Serverless）环境中的轻量级组件。

#### 架构分层

```
modules/collector/
├── cmd/moox-collector/main.go    # SCF 入口
├── internal/
│   ├── bootstrap/                 # 启动编排
│   ├── collector/                 # 采集器接口 + 注册中心
│   │   ├── collector.go           # Collector 接口定义
│   │   └── registry.go            # 全局注册中心（init() 自注册）
│   ├── executor/                  # 执行器（调度采集任务）
│   ├── heartbeat/                 # 心跳模块（动态获取采集规则）
│   ├── reporter/                  # 状态上报
│   ├── cloudfunction/             # SCF 云函数适配
│   ├── storage/                   # Storage 客户端（JSON-over-HTTP）
│   └── dns/                       # DNS 代理（交易所 IP 优选）
├── pkg/
│   └── binance/                   # Binance 采集器实现
│       ├── kline.go               # K 线采集
│       └── symbol.go              # 标的元数据采集
└── config/                        # 配置文件
```

#### 插件化采集器架构

- **接口定义**：`Collector` 接口定义统一的 `Collect()`、`Name()`、`Type()` 方法
- **自注册**：每个采集器实现通过 `init()` 函数自动注册到全局注册中心
- **当前实现**：Binance K 线采集器、Binance 标的采集器
- **扩展点**：配置中已预留 OKX、Twitter、CoinDesk、Ethereum 等数据源

#### 配置机制（三层）

1. **静态配置**：启动时从配置文件加载基础参数
2. **动态下发**：通过心跳从 Admin Server 获取采集规则（不持久化到本地）
3. **一致性校验**：使用 MD5 校验规则版本，避免重复加载

#### 与其他模块交互

| 交互方向 | 协议 | 说明 |
|----------|------|------|
| Collector → Admin | JSON-over-HTTP + HMAC | 心跳上报状态、获取采集规则 |
| Collector → Storage | JSON-over-HTTP | 三个 REST API 端点写入时序/记录数据 |
| Collector → 交易所 | HTTP/REST | DNS 优选 IP 访问交易所 API |

#### SCF 环境适配

- Panic 恢复：SCF 冷启动异常自动恢复
- 同步状态上报：每次采集完成后立即上报
- ServerInfo 动态更新：运行时环境信息动态采集
- K 线闭合处理：根据交易所时间判断 K 线是否闭合
- 并发控制：通过 channel + goroutine 池控制并发采集

#### DNS 优选机制

```
定时解析域名 → 多 IP 探测 → 延迟排序 → 缓存最优 IP
```

通过自建 DNS 代理优化对交易所 API 的访问成功率，避免 DNS 污染和延迟问题。

### 4.5 Web-Host（前端宿主网关）

**路径**：`web-host/`

Web-Host 是一个 Go 编写的**静态文件服务器 + API 反向代理网关**，将前端 `dist` 通过 statik 嵌入单个二进制文件。

#### 架构角色

```
浏览器 ──HTTP──▶ web-host (:8080)
                    │
          ┌─────────┼──────────────┐
          │                         │
    静态文件服务              API 反向代理
    (statik 嵌入)             │
                    ┌────────┼────────┐
                    │                 │
              Admin Gateway      Storage tRPC
              (:11000)           (:20200-20202)
```

#### 对外暴露

- `/api/admin/*` → 反向代理到 Admin Gateway
- `/api/storage/*` → 反向代理到 Storage tRPC 服务
- 其他路径 → 静态文件服务（ETag / Gzip / 缓存）

**优势**：部署时只需单个二进制文件，无需独立 Web 服务器。

## 5. 协议与通信设计

### RPC 框架

项目使用**腾讯 tRPC-Go**（非 gRPC），所有 proto package 使用 `trpc.*` 前缀。

### Proto 文件组织

共 11 个 `.proto` 文件，按模块隔离：

| 模块 | Proto 文件 | 定义内容 |
|------|-----------|---------|
| Storage | `common.proto` | 公共类型（Space, Dataset, Subject 等） |
| | `metadata.proto` | 元数据管理服务 |
| | `access.proto` | 数据读写服务 |
| | `primary_store.proto` | 内部主存服务 |
| | `data_view.proto` | 派生视图查询服务 |
| | `event.proto` | 领域事件定义 |
| Admin | `admin.proto` | 认证 + 用户/角色管理 |
| | `collect.proto` | 采集管理 + 节点管理 |
| | `infra.proto` | 包管理 + 云节点管理 |
| | `ops_service.proto` | 异步任务 + 监控 |

### 传输协议

| 场景 | 协议 | 说明 |
|------|------|------|
| 浏览器 → web-host | HTTP / JSON | 标准 RESTful |
| web-host → Admin | HTTP / JSON | 反向代理 |
| web-host → Storage | HTTP / JSON | 反向代理 |
| Admin 内部服务 | tRPC TCP / PB | 服务间二进制通信 |
| Collector → Admin | HTTP / JSON + HMAC | 采集器上报 |
| Collector → Storage | HTTP / JSON | 数据写入 |
| Storage 内部 | tRPC TCP / PB | Access → PrimaryStore |

### 网关架构

Admin 模块实现统一 HTTP 网关：

```
HTTP 请求
    │
    ▼
┌─────────────────────────────────────┐
│         统一网关 (Gateway)           │
│  ┌─────────────────────────────────┐│
│  │  路由匹配                        ││
│  │  /api/admin/{svc}/{method}      ││  ← JWT 鉴权
│  │  /api/service/{svc}/{method}    ││  ← HMAC 鉴权
│  └────────────┬────────────────────┘│
│               │                      │
│  ┌────────────▼────────────────────┐│
│  │  Filter 链                      ││
│  │  1. JWT 校验 / HMAC 校验         ││
│  │  2. 限流                         ││
│  │  3. Space ID 注入 (X-Space-Id)  ││
│  └────────────┬────────────────────┘│
│               │                      │
│  ┌────────────▼────────────────────┐│
│  │  SerializationTypeNoop 透传      ││
│  │  JSON Body → PB Message 转换     ││
│  └────────────┬────────────────────┘│
│               │                      │
│  ┌────────────▼────────────────────┐│
│  │  tRPC Service 调用               ││
│  └─────────────────────────────────┘│
└─────────────────────────────────────┘
```

网关使用 `SerializationTypeNoop` 纯透传 body，不参与序列化，由下游服务完成 JSON → PB 转换。

### Space 隔离

通过 `X-Space-Id` HTTP 头实现多空间数据隔离，网关层注入，业务层透明使用。

## 6. 数据架构

### 量化金融核心概念

| 概念 | 说明 |
|------|------|
| **DataSource** | 数据源（替代传统 Exchange 概念），如 `binance`、`okx` |
| **Subject** | 数据主体（如交易标的 `BTC`），不归属于特定 DataSource |
| **SubjectSymbol** | 外部代码映射，如 `BINANCE:BTCUSDT` → 统一 Subject `BTC` |
| **Factor** | 因子，不拆分 Def/Instance，统一为 Factor 概念 |
| **View** | 异步物化只读查询入口，支持时序 OLAP 和全文检索 |

### 数据流

```
    交易所 API                    其他数据源
        │                            │
        ▼                            ▼
┌───────────────┐           ┌───────────────┐
│   Collector    │           │  其他 Collector │
│  (SCF 部署)    │           │               │
└───────┬───────┘           └───────┬───────┘
        │                           │
        │  HTTP / JSON              │
        ▼                           ▼
┌───────────────────────────────────────────┐
│              Access Service                │
│            (Storage 模块)                  │
├───────────────────────────────────────────┤
│  写入链路:                                  │
│  校验 → 路由 → Pebble 主存 → 事件发布       │
│                                    │       │
│                              事件  ▼       │
│                          ┌───────────────┐ │
│                          │  NATS 事件总线 │ │
│                          └───────┬───────┘ │
│                                  │         │
│                          ┌───────▼───────┐ │
│                          │   Deriver     │ │
│                          │  (异步派生)    │ │
│                          └───┬───────┬───┘ │
│                              │       │     │
│                    ┌─────────▼─┐ ┌───▼───┐ │
│                    │  DuckDB   │ │ Bleve │ │
│                    │ (时序视图) │ │(全文)  │ │
│                    └───────────┘ └───────┘ │
│                                            │
│  查询链路:                                  │
│  Access → DataView → DuckDB / Bleve        │
└───────────────────────────────────────────┘
        ▲
        │ HTTP / JSON
        │
┌───────┴───────┐
│   Web-Host    │ ← 浏览器
│  (API 代理)    │
└───────────────┘
```

### 写入语义

- **列级 patch/upsert**：只更新传入的字段，未传入字段（NULL）不覆盖
- **TimeSeries**：按 `subject + freq + dimension` 索引，version = data_time
- **Record**：按 `record_id` 索引，version = record_version

## 8. 安全与鉴权

### 鉴权层级

```
浏览器 ──JWT──▶ web-host ──透传──▶ Admin Gateway
                                           │
                                    ┌──────┴──────┐
                                    │ JWT 校验    │ ← 用户请求
                                    │ HMAC 校验   │ ← 内部服务请求
                                    └─────────────┘
```

| 鉴权方式 | 适用场景 | 机制 |
|----------|----------|------|
| JWT | 浏览器 → 系统 | 登录获取 Token，网关校验有效性 |
| 盐值挑战 + HMAC | Collector → Admin | 动态盐值 + HMAC 签名，防重放 |
| 裸 HTTP 校验 | 内部服务 | IP 白名单 + 共享密钥 |

### 数据隔离

- **Space 隔离**：通过 `X-Space-Id` HTTP 头实现多空间数据隔离
- **行级权限**：基于 Role-Permission 模型的 RBAC 权限控制

## 10. 构建与发布

### Makefile 命令

| 命令 | 说明 |
|------|------|
| `make build` | 编译所有模块二进制到 `bin/` |
| `make test` | 运行测试 |
| `make deploy ARGS="..."` | 部署到目标环境 |
| `make release` | 打包发布 |
| `make proto` | 生成 proto 代码 |
| `make clean` | 清理构建产物 |

### 构建脚本

`scripts/` 目录下存放具体脚本，由 Makefile 调用：

- `build.sh` — 多平台编译（darwin/linux × amd64）
- `deploy.sh` — 部署到远程节点
- `release.sh` — 打包发布产物
- `proto.sh` — tRPC proto 代码生成
- 其他运维脚本

### 发布产物

`release/` 目录按平台和架构组织：

```
release/
├── deploy-stage/              # 部署暂存
```
